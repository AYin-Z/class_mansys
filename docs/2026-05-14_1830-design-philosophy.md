# 设计哲学 — class_mansys

## 项目作者反复坚持的几条设计原则

### 1. 渐进务实主义

项目从小程序 MVP 起步，资质受阻后迅速转 Android APK，代码几乎不改。不追求一刀切的完美架构，每步都解决一个真实障碍。表现：
- 核心业务逻辑与 UI 解耦充分，切换分发端只影响登录入口
- 条件编译仅用于平台差异，不用于业务逻辑分支
- 文档（`PROJECT_OVERVIEW.md`）明确记录每一次转向的原因

### 2. 免运维优先

设计上默认服务端是"黑盒"，客户端承担更多逻辑。体现：
- `request.ts` 的 401 自动跳转完全在客户端闭环
- App 内更新检查 (`update-checker.ts`) 走 CDN 版本号比对，不依赖后端
- 权限矩阵客户端缓存，页面内的 UI 显示/隐藏由前端判断

### 3. 透明审计优于手动日志

`operationLog` 中间件对所有写操作自动 INSERT 到 `operation_logs` 表——不要求各 Controller 手动调用日志函数。
这是系统中最像"产品级设计"的决定，与 MVP 基调形成鲜明对比。

### 4. 权限扁平化而非 RBAC

不搞角色继承、角色组、资源级权限树。9 种角色 × 15 种权限 = 一个扁平 `Record`。
够用就停，不追求架构抽象——符合项目实际诉求（一个区队 ≤ 40 人）。
