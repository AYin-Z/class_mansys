## 🔴 P0 — 必须修

### 1. 班费模块严重不完整（9页中仅2页真实）

这是整个 App 中唯一严重偏离 PRD 的模块。18 个模块中 17 个前后端完整对齐，但班费模块有 6/9 的页面使用纯硬编码 mock 数据。

**假的页面：**

| 页面 | 假数据表现 |
|------|-----------|
| `fee/collection.vue` | 收缴金额/已交人数/成员列表全 hardcode，refresh 按钮只弹 toast |
| `fee/apply-approve.vue` | 审批列表全本地 fake 数组，approve/reject 改本地状态 |
| `fee/publication.vue` | 报销公示数据全 hardcode |
| `fee/records.vue` | 收支记录数组全 hardcode，过滤器在本地 mock 上操作 |
| `fee/supervision.vue` | 余额/交易/审批轨迹全 hardcode |
| `fee/vote.vue` | 投票项/计数全本地 mock，与后端 vote 模块无关联 |

**后端能力与 UI 设计严重脱节：**

- UI 要求**四步审批流**（经办人→区队长初审→班干部投票≥2/3→执行报销），后端只有单步 approve
- UI 的投标表决（fee/vote）与后端独立的投票模块毫无关联
- 收缴功能无任何后端支撑（`expenses` 表是单条记录模式，不支持按人追踪）
- 凭证附件、明细字段在 Model 中没有对应

**改造预估：** 约需新增 3 张表（`fee_collections`、`fee_approval_chain`、`fee_publications`）+ 扩展现有 route/controller，周末 1-2 天可完工。

### 2. `src/utils/auth.js` 中的 `isAdminRole()` 恒为 false

比较的是中文字符串（"区队长"）与后端 INT 编码。所有通过此函数判断权限的旧页面权限形同虚设。新代码已统一走 `roles.ts` 的 `hasPermission()`。

### 3. JWT 权限检查两套标准

- `middleware/auth.js` 的 `authorizeAdmin` 检查 `req.user.isAdmin`
- 各 Controller 自己检查 `user.role`
- 若 JWT 签发时不带 `isAdmin` 字段，admin 路由全 403

## 整体评估

18 个模块中 **17 个前后端完整对应**。班费模块是唯一的重大缺口。

这是一个典型的 Vibe Coding 产物格局：大量 UI 页面先行设计好了（9 个页面），但核心业务逻辑（班费多级审批流程）没有被实际建模到后端。前端 pages 数量远多于后端实际支持的端点——这是 AI 辅助开发最容易出现的模式（模型更擅长写 UI 而不是设计数据库 schema）。

**PRD 完成度：约 85%**（18 模块中 17 个完整 + 1 个严重残缺）。
